Euro 2028 Predictor — Dark Theme Palette

Design artifact authored against the Stage 2 light palette in Design Direction and Components, the owner rulings in CLAUDE.md §5, and the 4.5:1 contrast requirement in both themes.

PROPOSED — NOT ADOPTED. The design programme governs nothing. This file changes no product code and no tokens.css value. Cite as “(design programme — proposed, not yet adopted).”

1. How the dark derivation works

The Direction doc already states the rule: “Dark theme mirrors every pairing at matching contrast — same roles, darker surfaces, brighter text.” That is exactly what this palette does, with four deliberate departures from naive inversion.

  1. Roles are preserved, values are not inverted. Each token keeps its single meaning (confirmed green, provisional amber, live red, locked slate, joker gold, info blue). A dark value is never the light value's mathematical inverse — inversion destroys hue identity and wrecks contrast. Each dark colour is re-authored at the same hue, then its lightness is chosen so the pairing clears 4.5:1.
  2. Surfaces get darker; the chrome gets lighter than the page. Naive inversion would make the navy header the lightest thing on screen. Instead the three dark surfaces step upward from the page — page #06111F → raised card #0E1F33 → navy chrome #0F2A4F. Chrome stays recognisably brand navy and stays the lightest of the three, so elevation still reads correctly. This makes chrome the binding surface for every light-on-dark text check, not the page — so every text token below is solved against chrome, the hardest case.
  3. Semantic “soft” surfaces are re-authored, not darkened tints. Light chips are saturated text on a pale tint of the same hue. Dark chips are bright text on a deep, desaturated tint of the same hue. Simply darkening #DCFCE7 yields a muddy green that fails against its own text; each dark soft surface is authored independently at ~10–18% lightness and its text is then solved upward until it clears 4.5:1 against the chip, the card and the page.
  4. Action blue climbs its own ramp; no new family is introduced. Light action blue #0284C7 and the ruled sky accent #5FC7F5/#38BDF8 are the same hue (≈199°) at different lightness — one sky ramp, not two blues. In dark, action blue becomes #38BDF8, a value already declared by the owner ruling. Nothing new is invented. See flag F4 for the consequence.

Closing the three §4 gaps

The Direction doc declares no border colour, no muted-text colour and no info colour, though §7 (chips), §11 (hairline dividers) and §13 (states) all require them. The concepts files improvised #D9E0EA and #7C879A. Both are authored properly here, for both themes.

  • Muted text. The improvised #7C879A reaches only 3.38:1 on the page surface — it fails the project's own requirement. Authored light muted is #5D6B80 (same 216° family, 5.04:1 on page). A separate text-muted-on-chrome (#99ABC2) exists because muted ink on navy chrome can never clear 4.5:1 in the light theme — see flag F3.
  • Border. One border colour cannot serve two jobs. Hairline dividers (§11) are decorative and carry no WCAG minimum; control boundaries (inputs, the score stepper, the outlined info chip) are UI components and owe 3:1 under WCAG 1.4.11. The palette therefore authors two: border-subtle for dividers and border-strong for boundaries. The improvised #D9E0EA is a fine divider (1.33:1) and an unusable control boundary.
  • Info. Authored in the sky ramp as an outlined chip, per §7's “neutral info (blue outline)”. Its border carries the signal, so the border itself clears 3:1 — the only chip border that must.

5. Flags for owner decision

None of the following is silently resolved. Each needs an owner ruling before adoption.

  1. F1 — The Stage 2 light palette fails 4.5:1 in seven pairings, across three declared tokens. This is not a dark-theme problem and is not fixed here, because these are declared values the brief told me to honour. The contrast audit enforces 4.5:1 with an empty allowlist, so adopting the light palette as written would fail the audit. Computed above:
    • Action blue #0284C7 — as link text 3.81:1 on page, 4.10:1 on card; and white on a filled action-blue button is 4.10:1. All three fail. This is the most consequential: it is the primary button and every link.
    • Joker gold #A6790A — 3.54:1 on its own chip and 3.92:1 as text on a white card. The worst offender in the palette.
    • Live red #DC2626 — 3.95:1 on its chip.
    • Provisional amber #B45309 — 4.49:1 on its chip, failing by 0.01.
    Every one of these clears 4.5:1 comfortably in the authored dark theme. Darkening each light text value by roughly 6–12% lightness fixes all seven without any hue change — but that edits declared Stage 2 colours, so it is an owner decision, not mine.
  2. F2 — Danger and live are near-identical in both themes, by inheritance. Light danger #B91C1C against light live #DC2626 is 1.34:1; the authored dark pair is 1.36:1. Dark is therefore faithful, not worse. The distinction survives because §7 requires icon + label on every chip and gives live a pulsing dot. If the owner wants the two reds separable by colour alone, one of them must move hue (danger toward crimson ≈350°, or live toward vermilion ≈8°) in both themes — a deviation from the declared palette, so it is not taken here.
  3. F3 — Muted text can never clear 4.5:1 on light navy chrome. Light muted #5D6B80 on brand navy #163B73 is 2.03:1 and no colour at that lightness can clear 4.5:1 against that surface. A separate text-muted-on-chrome (#99ABC2, 4.70:1) is authored for the light theme. In dark, chrome is light enough that the single muted token (#93A4B8, 5.63:1) works everywhere. The owner should confirm that a chrome-specific muted token is acceptable, or that muted text is simply banned on light chrome.
  4. F4 — In dark, the link colour and the broadcast accent are adjacent values on the same sky ramp. Dark action blue is #38BDF8 and the accent is #5FC7F5 — both declared by the owner ruling, ~2° apart in hue. They are hard to tell apart in isolation. No new accent family was introduced to separate them, per the constraint. Options: reserve the accent strictly for chrome/broadcast contexts where links never appear (recommended, and what the samples above assume), or declare a distinct link value inside the sky ramp.
  5. F5 — Info blue shares the sky hue with action blue (both ≈200°). §7 calls info a “blue outline” chip, so form (outline, not fill) rather than hue is what separates a static info chip from an interactive link. That reading is applied here and matches the product's existing --state-info: #0d659f. Confirm this is intended; the alternative is a second blue family, which the constraints forbid.
  6. F6 — The improvised concept values should be retired, not adopted. #7C879A (muted) fails at 3.38:1 and #D9E0EA (border) is unusable as a control boundary at 1.33:1, though it is a fine hairline divider. The concepts files should be re-rendered against the authored tokens before adoption.

6. Constraint compliance

  • Joker gold stays joker-exclusive. #EFC55C/#352810 appear only on the joker chip. No gold in chrome, tabs, marks or borders, in either theme.
  • No new accent families. Every colour is either a declared Stage 2 value, its same-hue dark counterpart, or the ruled sky pair. Maximum hue drift light→dark is 9° (provisional), and every other semantic drifts ≤5°.
  • Semantics keep one meaning each. Six roles, six colours, both themes.
  • Every text pairing clears 4.5:1 in dark — 0 failures across 24 checked pairings. The narrowest margins are the live chip at 5.54:1 and muted text on navy chrome at 5.63:1; every other dark pairing exceeds 6:1. See the self-check banner above, which recomputes this on load.
